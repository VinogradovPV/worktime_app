# Анализ проблемы синхронизации с Yandex Cloud PostgreSQL

## Статус подключения

**Текущее состояние:** TCP-соединение устанавливается, но PostgreSQL не принимает данные корректно.

---

## Диагностика проблемы

### Что работает ✅
- PostgreSQL запущен на `158.160.165.2:5432` и слушает на `0.0.0.0:5432`
- TCP 3-way handshake успешен (соединение устанавливается)
- Security Group в Yandex Cloud разрешает TCP порт 5432 с `0.0.0.0/0`
- `pg_hba.conf` правильно настроен: `host all all 0.0.0.0/0 md5`
- Docker контейнер работает, DNAT-правило создано (`11 пакетов` прошло)
- iptables правила добавлены для `br-7b9ea17ac129`
- Локальное подключение работает: `docker exec sync-db psql -U sync_user -d sync_database -c "SELECT 1"` возвращает результат

### Что не работает ❌
- PostgreSQL не отвечает на startup message (SSL request или plain)
- Соединение закрывается сервером после 3-way handshake
- Нет записей в логах PostgreSQL о попытках подключения
- Непоследовательное поведение: иногда `0 байт` (FIN), иногда `timeout`

## Возможные причины

### 1. **Асимметричная маршрутизация** (ВЕРОЯТНО)
- Пакеты от нас доходят до сервера (SYN → SYN-ACK работает)
- Ответные пакеты (DATA) теряются на обратном пути
- Признак: TCP handshake успешен, но данные не передаются

**Решение:**
```bash
# На сервере проверить маршруты
ip route show
ip rule show

# Проверить что ответные пакеты идут через правильный интерфейс
sudo tcpdump -i any port 5432 -n
```

### 2. **Проблема с conntrack/nf_conntrack** 
- iptables conntrack не отслеживает соединение правильно
- Ответные пакеты отбрасываются как "invalid"

**Решение:**
```bash
# Проверить conntrack таблицу
sudo conntrack -L | grep 5432

# Увеличить лимиты conntrack
sudo sysctl -w net.netfilter.nf_conntrack_max=1000000
sudo sysctl -w net.netfilter.nf_conntrack_tcp_timeout_established=600
```

### 3. **Docker bridge не пробрасывает трафик правильно**
- NAT-правило создано, но bridge-интерфейс не маршрутизирует ответные пакеты
- Возможно, нужно явно разрешить FORWARD для ответных пакетов

**Решение:**
```bash
# Проверить FORWARD цепочку
sudo iptables -L FORWARD -n -v

# Добавить явное правило для ответных пакетов
sudo iptables -I FORWARD 1 -m conntrack --ctstate ESTABLISHED,RELATED -j ACCEPT
```

### 4. **Проблема с SSL/TLS на уровне ОС**
- Ядро Linux отбрасывает пакеты с SSL-флагами
- Возможно, нужно отключить offloading

**Решение:**
```bash
# Проверить offloading
ethtool -k eth0 | grep -i tls

# Отключить если включено
sudo ethtool -K eth0 tls-hw-tx-offload off
sudo ethtool -K eth0 tls-hw-rx-offload off
```

### 5. **Проблема с Yandex Cloud VPC/VPN**
- Виртуальная машина в Yandex Cloud может иметь ограничения на входящий трафик
- Возможно, нужно настроить маршруты через VPN-интерфейс `wg-yandex`

**Решение:**
```bash
# Проверить VPN-интерфейс
ip link show wg-yandex
ip addr show wg-yandex

# Проверить маршруты через VPN
ip route show | grep wg-yandex
```

## Пути решения (в порядке приоритета)

### Вариант 1: Использовать Managed Service for PostgreSQL в Yandex Cloud
**Преимущества:**
- Полностью управляемый сервис
- Встроенная поддержка SSL
- Автоматические резервные копии
- Масштабируемость

**Недостатки:**
- Платный сервис (примерно $10-50/месяц)
- Требует переноса данных

**Действия:**
1. Создать Managed Service for PostgreSQL в Yandex Cloud
2. Добавить IP `86.97.182.233` (или диапазон) в Security Group
3. Обновить connection string в приложении

---

### Вариант 2: Использовать SSH-туннель для подключения
**Преимущества:**
- Работает с любым PostgreSQL
- Безопасное шифрованное соединение
- Не требует изменений на сервере

**Недостатки:**
- Небольшое снижение производительности
- Требует SSH-ключа

**Действия:**
```bash
# На сервере разрешить SSH
ssh -i key.pem -L 5432:localhost:5432 user@158.160.165.2

# В приложении подключаться к localhost:5432
```

---

### Вариант 3: Использовать HTTP/REST API вместо прямого подключения к БД
**Преимущества:**
- Работает через любой файрвол
- Можно добавить аутентификацию и rate-limiting
- Легче масштабировать

**Недостатки:**
- Требует написания API на сервере
- Небольшое снижение производительности

**Действия:**
1. Создать REST API на Node.js/Express на сервере
2. Endpoints для CRUD операций с workdays
3. Подключаться из приложения через HTTP

---

### Вариант 4: Диагностика и исправление текущей настройки
**Преимущества:**
- Работает текущая инфраструктура
- Минимальные затраты

**Недостатки:**
- Требует глубокого анализа сетевой конфигурации
- Может быть сложно отладить

**Действия:**
1. Запустить `tcpdump` на сервере и снять трассировку трафика
2. Проверить conntrack таблицу
3. Проверить маршруты и интерфейсы
4. Отключить SSL offloading если включено
5. Проверить логи ядра (`dmesg`)

---

## Рекомендация

**Для быстрого решения:** Вариант 1 (Managed Service for PostgreSQL) или Вариант 3 (REST API).

**Для минимальных затрат:** Вариант 4 (диагностика) + Вариант 2 (SSH-туннель как fallback).

---

## Следующие шаги

1. **Немедленно:** Запустить `tcpdump` и снять трассировку трафика
2. **Параллельно:** Проверить conntrack и маршруты
3. **Если не поможет:** Переходить на Вариант 1 или 3
