# TODO: регистрация, авторизация и ролевая модель Worktime

## 0. Цель задачи

Реализовать MVP регистрации, авторизации, справочников структурных подразделений и должностей, а также ролевую модель доступа:

- `user` — видит только свои данные;
- `unit_manager` — видит себя и сотрудников своей инспекции / отдела;
- `department_manager` — видит себя, департамент и подчиненные инспекции / отделы;
- `admin` — имеет глобальный доступ, управляет пользователями, справочниками, ролями и может редактировать чужие рабочие события.

Проверка email/SMS не используется. Новые пользователи проходят регистрацию через заявку и получают доступ только после подтверждения администратором.

## 15. Тестовые сценарии

### 15.1. Регистрация

- [ ] Пользователь может открыть форму регистрации.
- [ ] Пользователь не видит выбор роли.
- [ ] Пользователь выбирает подразделение и должность из справочников.
- [ ] Пользователь не может зарегистрироваться с занятым логином.
- [ ] Пользователь не может зарегистрироваться с коротким паролем.
- [ ] Пользователь не может зарегистрироваться, если пароль и подтверждение не совпадают.
- [ ] После регистрации пользователь получает `pending`.
- [ ] Пользователь с `pending` не может войти в приложение.

### 15.2. Администратор

- [ ] Первый администратор создается seed-скриптом.
- [ ] Администратор может войти.
- [ ] Администратор видит заявки на регистрацию.
- [ ] Администратор может подтвердить заявку.
- [ ] Администратор может отклонить заявку.
- [ ] Администратор может назначить роль.
- [ ] Администратор может указать `managedOrgUnitId`.
- [ ] Администратор может сбросить пароль.
- [ ] Администратор может управлять справочниками.
- [ ] Администратор может редактировать чужие рабочие события.

### 15.3. User

- [ ] `user` видит только свои данные.
- [ ] `user` не может получить чужие данные через API.
- [ ] При попытке получить чужие данные API возвращает `403`.

### 15.4. Unit manager

- [ ] `unit_manager` видит себя.
- [ ] `unit_manager` видит сотрудников своего подразделения.
- [ ] `unit_manager` не видит сотрудников других подразделений.
- [ ] `unit_manager` может формировать отчеты по своему подразделению.
- [ ] `unit_manager` не может редактировать чужие события.

### 15.5. Department manager

- [ ] `department_manager` видит себя.
- [ ] `department_manager` видит сотрудников своего департамента.
- [ ] `department_manager` видит сотрудников подчиненных подразделений.
- [ ] `department_manager` не видит другие департаменты.
- [ ] `department_manager` может формировать отчеты по департаменту.
- [ ] `department_manager` может формировать отчеты в разрезе инспекций / отделов.
- [ ] `department_manager` не может редактировать чужие события.

### 15.6. Access control

- [ ] Все ограничения ролей проверяются на backend.
- [ ] Недоступные запросы возвращают `403`.
- [ ] Нельзя получить чужие данные подбором `userId`.
- [ ] Нельзя получить чужой отчет подбором `orgUnitId` или `departmentId`.
