# Руководство по синхронизации API

Инструкция по настройке API-сервера синхронизации Worktime
Для администратора сервера. Без команд для ИИ-агентов, Codex или Мануса.
Версия: MVP API + PostgreSQL в Docker + Nginx + HTTPS
0. Что делает эта инструкция
Цель: заменить прямое подключение мобильного приложения к PostgreSQL на безопасную схему синхронизации через HTTPS API. Мобильное приложение больше не должно подключаться к порту 5432 и не должно хранить логин/пароль базы данных.
Главная идея: мобильное приложение отправляет события рабочего времени в API. API валидирует события, сохраняет их в PostgreSQL и возвращает клиенту подтвержденные данные. PostgreSQL остается закрытым от публичного интернета.
1. Итоговая архитектура
Снаружи открыт только HTTPS порт 443. На время выпуска сертификата открыт порт 80.
PostgreSQL не публикуется наружу и доступен только внутри Docker network.
API принимает события от мобильного приложения, сохраняет их в БД и пересчитывает итоги рабочего дня.
Синхронизация работает по принципу local-first: приложение сохраняет событие локально, затем отправляет его на сервер.
Если API временно недоступен, приложение продолжает работать локально и синхронизируется позже.
2. Предварительные условия
Сервер Ubuntu 22.04 или 24.04 в Yandex Cloud или на другой Linux VM.
Есть SSH-доступ к серверу с sudo-правами.
Есть домен или поддомен для API, например api.example.ru. Для теста можно временно работать по IP и HTTP, но для мобильного приложения нужен HTTPS.
DNS A-запись домена указывает на публичный IP сервера.
В Yandex Cloud Security Group разрешены входящие 80 и 443. SSH 22 лучше разрешить только с IP администратора.
Публичный доступ к PostgreSQL 5432 должен быть закрыт.
3. Настройка Security Group в Yandex Cloud
Откройте Yandex Cloud Console.
Перейдите в Virtual Private Cloud -> Security groups.
Откройте security group, которая назначена сетевому интерфейсу VM.
В разделе Ingress добавьте или оставьте правила для 80/tcp и 443/tcp от 0.0.0.0/0.
Для SSH 22/tcp укажите только ваш административный IP, если это возможно.
Удалите правило 5432/tcp от 0.0.0.0/0. PostgreSQL не должен быть доступен из интернета.
Проверьте, что egress-трафик разрешен, чтобы сервер мог скачивать пакеты и выпускать сертификат.
Важно: если на сервере дополнительно включен ufw или iptables, правила Yandex Cloud не заменяют локальный firewall. Проверяйте оба уровня.

15. Как мобильное приложение должно синхронизироваться с API
Минимальный алгоритм клиента:
Пользователь нажимает кнопку Начать / Перерыв / Продолжить / Выйти / Вернуться / Завершить.
Приложение сразу сохраняет событие локально, чтобы работа не зависела от сети.
Событие получает client_event_id и статус pending_sync.
При наличии сети приложение отправляет события на POST /api/v1/sync/events.
Сервер делает idempotent upsert по user_id + client_event_id.
Сервер пересчитывает рабочий день и возвращает summary.
Клиент обновляет локальный день полученным summary.
Если сеть недоступна, событие остается локально и отправляется позже.

20. Минимальные правила безопасности
Не открывайте PostgreSQL наружу.
Не храните пароль PostgreSQL в мобильном приложении.
Используйте HTTPS для всех запросов приложения.
API token храните как секрет и периодически меняйте.
Для production замените общий API token на нормальную авторизацию пользователей и устройств.
Ограничьте SSH по IP администратора.
Делайте резервные копии БД.
Не логируйте полные токены и персональные данные.
Следите за объемом postgres-data и backups.
21. Что улучшить после MVP
Перейти с общего Bearer token на авторизацию пользователей: login, refresh tokens, device sessions.
Добавить таблицы users, organizations, devices и роли доступа.
Добавить миграции через Alembic вместо автоматического CREATE TABLE в коде.
Добавить rate limiting на Nginx или API.
Добавить централизованный audit log ручных правок.
Добавить полноценные отчеты на сервере: день, неделя, месяц, год.
Перенести PostgreSQL в Yandex Managed Service for PostgreSQL, когда MVP стабилизируется.
Для Managed PostgreSQL использовать настройки Yandex Cloud: security groups, SSL и порт managed-кластера.
